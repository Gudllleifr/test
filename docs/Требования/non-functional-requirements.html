Нефункциональные требования

Документ содержит нефункциональные требования к API Contract Hub (MVP v1.0). Предназначен для команды разработки, тестировщиков, архитектора и технических руководителей.

Матрица покрытия категорий NFR​

Производительность​

Система обслуживает ~110 пользователей в рамках одной организации. Пиковая нагрузка — утро и после обеда. Медленная система увеличивает риск низкого внедрения (бизнес-риск 40%).

NFR-001 — Время отклика API​

Серверные API-endpoints должны отвечать в установленные временные рамки при нормальной нагрузке (≤ 20 одновременных пользователей).

ОперацияЦелевой показательМетрика
Получение списка сервисов< 500 мс (p95)95% запросов за 500 мс
Загрузка детали сервиса с историей< 800 мс (p95)95% запросов за 800 мс
Валидация и сохранение OpenAPI-файла< 2000 мс (p95)95% запросов за 2 с
Анализ критических изменений (oasdiff)< 3000 мс (p95)95% сравнений за 3 с
Загрузка данных графа зависимостей< 1000 мс (p95)95% запросов за 1 с

Приоритет: Must Have. Способ измерения: нагрузочное тестирование с k6, мониторинг p95 latency через Prometheus + Grafana.

NFR-002 — Рендеринг графа зависимостей​

СценарийЦелевой показательУсловие
Первичный рендеринг (≤ 60 узлов)< 2000 мсПилотный этап (2 команды)
Первичный рендеринг (≤ 120 узлов)< 5000 мсПолная экосистема (production)
Интерактивность60 FPS (минимум 45 FPS)Chrome DevTools

Приоритет: Must Have. Способ измерения: Lighthouse performance audit + ручное тестирование с 120 узлами.

NFR-003 — Пропускная способность​

Расчётная пиковая нагрузка: 110 пользователей × 20% активных в пик = 22 пользователя. Средняя сессия: 10 мин., 15 запросов. RPS = (22 × 15) / 600 ≈ 0,55 RPS. С учётом всплесков (×5): ~3 RPS.

Требование: система стабильно работает при нагрузке до 20 RPS (запас ×6 от расчётной) без деградации времени ответа.

Приоритет: Should Have. Способ измерения: k6 нагрузочный тест (20 виртуальных пользователей, 5 минут).

Надёжность и доступность​

Если система недоступна, разработчики не могут проверить безопасность изменений API — это увеличивает вероятность инцидентов.

NFR-004 — Доступность системы​

ПараметрЗначениеКомментарий
Целевой uptime (рабочие часы)≥ 99,0%≤ 1,7 ч простоя/мес. в рабочие часы
Минимальный uptime (24/7)≥ 97,0%≤ 21,6 ч простоя/мес.
Плановое обслуживаниеНерабочее времяОповещение команды за 24 ч
Время восстановления (RTO)< 2 чПосле обнаружения проблемы

Приоритет: Must Have. Способ измерения: Prometheus Alertmanager, оповещение при простое > 5 мин.

NFR-005 — Целостность данных​

  • Транзакционность: операции сохранения контракта выполняются в транзакции. Частично сохранённый контракт невозможен.
  • Бэкап: ежедневная резервная копия БД, хранение последних 7 копий.
  • RTO: восстановление из бэкапа ≤ 4 часов.
  • RPO: потеря данных не более чем за 24 часа (период между бэкапами).

Приоритет: Must Have. Потеря истории контрактов критична — это основная ценность системы.

Безопасность​

API Contract Hub хранит OpenAPI-контракты всех сервисов организации — фактически, архитектурную карту системы. Утечка может помочь злоумышленникам в разведке уязвимостей.

NFR-006 — Аутентификация и авторизация​

РольЧтениеЗагрузка/редактированиеАдминистративные функции
Гость (QA, Тех. руководитель)✓——
Разработчик✓✓ (свои сервисы)—
Архитектор✓✓ (все сервисы)Частично
Администратор✓✓ (полный доступ)✓
  • Аутентификация: JWT-токены (HS256). Access token — 8 часов, Refresh token — 7 дней.
  • Пароли: bcrypt-хэши, минимальная длина — 12 символов.
  • Защита от перебора: блокировка аккаунта на 15 мин. после 5 неудачных попыток; блокировка IP на 15 мин. после 20 попыток с одного адреса.

Приоритет: Must Have. SAML/LDAP/OAuth (SSO) запланированы на v2.0.

NFR-007 — Безопасность передачи и хранения данных​

  • Транспортный уровень: все endpoints доступны только по HTTPS (TLS 1.2+). HTTP-запросы перенаправляются на HTTPS.
  • Заголовки безопасности: Content-Security-Policy, X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security обязательны для всех ответов.
  • Защита от OWASP Top 10: валидация входных данных на бэкенде, защита от SQL-инъекций через Prisma ORM, санация вывода против XSS.
  • Аудит-лог: все операции записи логируются с user_id, timestamp, IP-адресом.

Приоритет: Must Have.

Масштабируемость​

NFR-008 — Архитектурная готовность к масштабированию​

ПараметрMVP (июнь 2026)Production (декабрь 2026)
Зарегистрированных сервисов≤ 30 (пилот)≤ 150
Версий контрактов (общее)≤ 300≤ 1 500
Одновременных пользователей≤ 10≤ 30
Рёбер в графе зависимостей≤ 100≤ 500
Объём БД (контракты)≤ 500 МБ≤ 5 ГБ

Docker Compose для MVP (одна машина), Kubernetes при росте нагрузки. PostgreSQL с GIN-индексами для JSONB, connection pooling через Prisma (макс. 20 соединений).

Приоритет: Should Have.

Удобство использования​

Ключевой бизнес-риск — низкая степень внедрения (вероятность 40%). Если система неудобна, разработчики перестанут ею пользоваться.

NFR-009 — Онбординг нового пользователя​

Новый пользователь (знакомый с REST API и OpenAPI) должен самостоятельно зарегистрировать первый сервис и просмотреть граф зависимостей не более чем за 30 минут.

Дополнительные требования:

  • встроенные подсказки для ключевых действий;
  • информативные сообщения при пустых состояниях (не пустой экран, а призыв к действию);
  • все сообщения об ошибках на русском языке, с описанием проблемы и следующим шагом.

Приоритет: Must Have. Способ измерения: юзабилити-тест с 3–5 разработчиками, не участвовавшими в разработке.

NFR-010 — Кроссбраузерность​

БраузерПоддержкаВерсия

Mozilla FirefoxПолная≥ 110

Минимальное разрешение: 1366×768. Время загрузки первой страницы (LCP): ≤ 3 сек. на скорости 10 Мбит/с.

Приоритет: Should Have.

Сопровождаемость​

NFR-011 — Качество кода и архитектурные стандарты​

  • Покрытие тестами: unit-тесты для бизнес-логики — ≥ 70% веток; интеграционные тесты для API-endpoints — ≥ 60% endpoints.
  • Статический анализ: ESLint и Prettier, проверяются в CI/CD pipeline.
  • Документация API: все endpoints документированы в OpenAPI 3.0, обновляются при каждом изменении.
  • Онбординг разработчика: новый разработчик поднимает окружение и запускает проект локально за ≤ 30 минут по инструкции в README.

Приоритет: Should Have.

Наблюдаемость​

NFR-012 — Логирование, метрики и оповещения​

ТипТребованияИнструмент
ЛогиСтруктурированные JSON, уровни WARN/INFO/DEBUG. Ротация 30 дней. Запросы к API логируются (method, path, status, duration, user_id). Чувствительные данные не логируютсяWinston + Loki/ELK
МетрикиHTTP request rate, response time (p50/p95/p99), error rate, DB connection pool, oasdiff execution timePrometheus + Grafana
Бизнес-метрикиКоличество сервисов, количество сравнений, количество обнаруженных критических измененийPrometheus custom metrics

Приоритет: Should Have. Без метрик невозможно доказать достижение KPI (-70% инцидентов к январю 2027).

NFR-013 — Совместимость с OpenAPI и экосистемой​

Тип совместимостиУровеньДетали
OpenAPI 3.1.xПолнаяРекомендуемая версия для новых сервисов
OpenAPI 3.0.xПолнаяТекущий стандарт большинства сервисов
Swagger 2.0 (OpenAPI 2.0)ПолнаяУстаревшие сервисы организации
YAMLПолнаяПредпочтительный формат для читаемости
JSONПолнаяГенерируемые спецификации часто в JSON
gRPC / Protocol BuffersНе поддерживаетсяВерсия 4.0
GraphQL SDLНе поддерживаетсяВерсия 4.0

Интеграция с PagerDuty: тег api-integration в инцидентах используется для измерения KPI. В MVP — только чтение через API PagerDuty для метрик дашборда.

Приоритет: Must Have. Без поддержки OpenAPI 2.0/3.0/3.1 система не сможет работать с реальными сервисами организации.
