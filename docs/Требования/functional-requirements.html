Функциональные требования

Документ содержит функциональные требования к первой версии продукта API Contract Hub (MVP v1.0). Предназначен для команды разработки, тестировщиков, архитектора и технических руководителей.

Скоуп требований

Требования распространяются исключительно на MVP v1.0. Границы MVP определены в Концепции и границах продукта.

Контекст системы​

ПараметрЗначение
ОрганизацияООО «ПринтСтройМонтажСервис»
Масштаб120 микросервисов, 15 команд, ~110 пользователей
ПлатформаNode.js + Express.js, React.js + React Flow, PostgreSQL
РазвёртываниеDocker Compose (MVP), Kubernetes (production)
Основная проблема18 API-инцидентов/мес., 4 часа ручного анализа на каждое изменение

UC-001 — Регистрация нового сервиса и загрузка контракта​

ПолеЗначение
IDUC-001
АкторыРазработчик серверной части (основной), Архитектор (дополнительный)
ЦельЗарегистрировать новый микросервис и загрузить его OpenAPI-спецификацию, чтобы другие команды могли видеть его контракт и зависимости
ПредусловияПользователь аутентифицирован (роль: разработчик или архитектор); подготовлен файл OpenAPI (JSON или YAML, версия 2.0/3.0/3.1); сервис ещё не существует в реестре

Основной поток​

  1. Пользователь переходит в «Реестр сервисов» и нажимает «Зарегистрировать сервис».
  2. Система отображает форму регистрации.
  3. Пользователь заполняет обязательные поля: название сервиса (3–100 символов), команда-владелец, контактный email, домен сервиса (payments, orders, users, logistics и др.) и опциональную ссылку на репозиторий.
  4. Пользователь загружает OpenAPI-файл: через выбор файла с устройства (.json или .yaml, макс. 5 МБ) или вставку содержимого в текстовое поле.
  5. Система выполняет валидацию: синтаксис (корректный JSON/YAML), соответствие схеме OpenAPI 2.0/3.0/3.1, наличие обязательных полей (info.title, info.version).
  6. Система отображает краткое резюме распознанного контракта: название API, версия, количество endpoints.
  7. Пользователь нажимает «Зарегистрировать».
  8. Система сохраняет сервис и контракт, устанавливает версию контракта как v1.
  9. Система перенаправляет на страницу детальной информации о сервисе.

Альтернативный поток A1 — загрузка новой версии контракта​

Пользователь открывает страницу существующего сервиса и нажимает «Загрузить новую версию контракта». Система пропускает форму регистрации и предлагает сразу загрузить файл. После валидации показывает автоматическое сравнение с предыдущей версией (вызов логики UC-002). После подтверждения сохраняет контракт как следующую версию (v2, v3 и т.д.).

Исключительные потоки​

E1 — файл не прошёл валидацию: система отображает список ошибок с указанием строки и типа ошибки (например: «Строка 42: поле responses обязательно для операции GET /users»). Сервис не сохраняется, пользователь остаётся на форме.

E2 — имя сервиса занято: система сообщает: «Сервис с именем [Имя] уже зарегистрирован. Если это ваш сервис, перейдите на его страницу для загрузки новой версии».

Бизнес-правила​

IDПравило
BR-001Имя сервиса уникально в рамках системы, регистр не учитывается (USERS-SERVICE = users-service)
BR-002Максимальный размер загружаемого файла — 5 МБ
BR-003Поддерживаемые версии: OpenAPI 2.0 (Swagger), 3.0.x, 3.1.x
BR-004Версия контракта назначается автоматически (v1, v2, …vN). Ручная установка версии не предусмотрена
BR-005История версий хранит последние 10 версий контракта (ограничение MVP)

UC-002 — Сравнение версий контракта​

ПолеЗначение
IDUC-002
АкторыРазработчик (основной), Архитектор (дополнительный)
ЦельПолучить детальный отчёт о различиях между двумя версиями контракта с классификацией изменений, чтобы оценить безопасность изменений API до деплоя
ПредусловияПользователь аутентифицирован; для сервиса существует минимум 2 версии контракта

Основной поток​

  1. Пользователь открывает страницу сервиса и переходит в «История версий».
  2. Пользователь выбирает «Базовую» (старую) и «Целевую» (новую) версию из выпадающих списков.
  3. Пользователь нажимает «Сравнить».
  4. Система выполняет анализ через библиотеку oasdiff: сравниваются схемы, endpoints, параметры, типы данных, коды ответов.
  5. Система классифицирует изменения: 🔴 критические (удаление endpoint, обязательного параметра, изменение типа), 🟠 устаревшие (deprecated), 🟢 новые (новые endpoints, необязательные параметры), 🟡 изменённые (описания, примеры).
  6. Система отображает: сводную панель с количеством изменений по категориям и параллельное сравнение двух колонок с подсветкой.

Альтернативный поток A1 — сравнение при загрузке​

При загрузке новой версии (поток A1 в UC-001) система автоматически запускает сравнение с предыдущей версией. Если обнаружены критические изменения, показывает предупреждение: «Обнаружено N критических изменений. Они могут нарушить работу зависимых сервисов. Продолжить сохранение?»

Бизнес-правила​

IDПравило
BR-006Изменения рассматриваются относительно «Базовой» (старой) версии
BR-007Классификация критических изменений выполняется по правилам oasdiff. Версия библиотеки зафиксирована в package.json и обновляется только после проверки совместимости
BR-008Одновременно сравниваются только 2 версии. Сравнение 3+ версий — вне скоупа MVP
BR-009Результаты сравнения не кэшируются в MVP — каждый запрос выполняет новый анализ

UC-003 — Просмотр графа зависимостей​

ПолеЗначение
IDUC-003
АкторыРазработчик, Архитектор, Технический руководитель
ЦельПолучить визуальное представление зависимостей между микросервисами, чтобы быстро понять архитектуру, найти критические пути и оценить масштаб влияния изменений
ПредусловияВ системе зарегистрировано минимум 2 сервиса с хотя бы одной зависимостью

Основной поток​

  1. Пользователь переходит в «Граф зависимостей» из главного меню.
  2. Система загружает данные обо всех сервисах и зависимостях из БД.
  3. Система рендерит интерактивный направленный граф: каждый сервис — узел (название + команда), зависимость «A использует API B» — стрелка от A к B.
  4. Пользователь взаимодействует с графом: масштабирование (колесо мыши), панорамирование (перетаскивание), наведение на узел (всплывающая подсказка), клик на узел (переход на страницу сервиса).
  5. Пользователь применяет фильтр по команде — граф показывает только её сервисы и зависимости.

Бизнес-правила​

IDПравило
BR-010Граф направленный. Стрелка от «потребителя» к «поставщику» API
BR-011В MVP зависимости регистрируются только вручную (автоматическое обнаружение — версия 3.0)
BR-012Один сервис может иметь неограниченное количество входящих и исходящих зависимостей
BR-013Граф отображает все сервисы, включая не имеющие зависимостей (изолированные узлы)

UC-004 — Анализ влияния изменений​

ПолеЗначение
IDUC-004
АкторыРазработчик (основной), Архитектор (дополнительный)
ЦельОпределить, какие сервисы пострадают от планируемых изменений API, получить контакты их владельцев и оценить масштаб влияния
ПредусловияСуществует сервис с минимум 2 версиями контракта; зарегистрированы входящие зависимости

Основной поток​

  1. Пользователь находится на экране результатов сравнения (после UC-002) или на странице сервиса.
  2. Пользователь нажимает «Анализ влияния» (доступна при наличии критических или изменённых endpoints).
  3. Система определяет список изменившихся endpoints из результатов сравнения.
  4. Система находит все сервисы, у которых в зависимостях указан целевой сервис.
  5. Система формирует отчёт: для каждого зависимого сервиса — название, команда-владелец, контактный email, перечень изменённых endpoints, тип изменения.
  6. Система показывает сводную статистику: «Затронуто N сервисов, из них M с критическими изменениями».
  7. Пользователь может скопировать список email-адресов владельцев для оповещения.

Бизнес-правила​

IDПравило
BR-014Анализ влияния основывается только на вручную зарегистрированных зависимостях (MVP). Точность ограничена полнотой данных
BR-015В MVP система не выполняет автоматическую оценку критичности для каждого зависимого сервиса — показывается только факт использования изменившихся endpoints. Это должно быть явно отражено в UI

UC-005 — Просмотр истории версий контракта​

ПолеЗначение
IDUC-005
АкторыРазработчик (основной), Архитектор (дополнительный), QA-инженер (только просмотр)
ЦельПолучить хронологический список всех версий контракта с возможностью просмотра, сравнения и отката
ПредусловияДля выбранного сервиса существует минимум 1 версия контракта

Основной поток​

  1. Пользователь открывает страницу сервиса и переходит на вкладку «История версий».
  2. Система отображает таблицу версий в обратном хронологическом порядке (новые — сверху): номер версии, дата и время загрузки, имя пользователя, краткая сводка изменений относительно предыдущей версии.
  3. Пользователь нажимает «Просмотреть» — система открывает полный текст спецификации с подсветкой синтаксиса.
  4. Пользователь выбирает две версии и нажимает «Сравнить» — запускается сценарий UC-002.

Альтернативный поток A1 — откат к предыдущей версии​

Доступен только для роли «Архитектор». Пользователь нажимает «Сделать активной» для одной из версий. Система показывает предупреждение: «Это действие создаст новую версию vM с содержимым vN. Продолжить?» После подтверждения — создаётся новая версия, оригинальная история остаётся неизменной.

Бизнес-правила​

IDПравило
BR-017История ограничена последними 10 версиями. При добавлении 11-й самая старая архивируется (метаданные хранятся, но версия недоступна для сравнения)
BR-018Удаление версии контракта не предусмотрено в MVP
BR-019Откат создаёт новую версию — история версий неизменна
