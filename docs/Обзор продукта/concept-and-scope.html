Краткое изложение​

API Contract Hub — платформа для управления жизненным циклом контрактов программных интерфейсов в микросервисных архитектурах. Проект направлен на решение критической проблемы несогласованных изменений API между сервисами, которая приводит к производственным инцидентам.

Система предоставляет централизованный реестр OpenAPI-контрактов, автоматический анализ критических изменений, визуализацию зависимостей между сервисами и раннее обнаружение несовместимых изменений до развёртывания в продуктивной среде.

Целевая аудитория: разработчики серверной части (~80 чел.), архитекторы (~8 чел.), инженеры DevOps (~12 чел.), технические руководители (~10 чел.).

MVP реализуется за 16–20 недель. Успех измеряется тремя ключевыми целями: снижение API-инцидентов на 70%, сокращение времени анализа влияния на 87,5%, покрытие 62% микросервисов в первый год.

Бизнес-требования​

Описание проблемы​

В ООО «ПринтСтройМонтажСервис» функционируют 120 микросервисов, взаимодействующих через REST API. За последние 6 месяцев (июль 2025 — январь 2026) произошло 108 производственных инцидентов, связанных с несовместимостью API — в среднем 18 инцидентов в месяц.

Типичный сценарий проблемы: команда обновляет API сервиса, не зная, кто его использует. После развёртывания возникают сбои в зависимых сервисах.

Текущие практики неэффективны по трём причинам:

  • ручной поиск зависимостей в исходном коде занимает 3–4 часа на каждое изменение;
  • документация контрактов быстро устаревает и не отражает реальное состояние системы;
  • координация между 15 командами разработки затруднена из-за отсутствия единого источника информации.
Важно

Количество микросервисов выросло с 80 до 120 за год и продолжит расти до 150 к концу 2026. При линейном росте прогнозируется увеличение инцидентов до 25–30 в месяц.

Критерии успеха MVP​

  • 10+ сервисов зарегистрированы с реальными OpenAPI-спецификациями.
  • Успешная демонстрация полного цикла: загрузка контракта → сравнение версий → обнаружение 3+ критических изменений.
  • Граф зависимостей корректно отображает связи между зарегистрированными сервисами.
  • Система функционирует на локальном окружении Docker Compose без критических ошибок.

Заявление о концепции продукта​

API Contract Hub трансформирует культуру разработки от «страха сломать что-то» к «уверенности в изменениях». Каждый разработчик будет иметь полную уверенность в безопасности своих изменений API, видеть зависимости так же ясно, как структуру кода в IDE, а критические изменения будут обнаруживаться автоматически за секунды, а не часы.

Границы и ограничения​

Основные функции системы​

Централизованный реестр контрактов API — единое хранилище всех версий OpenAPI-спецификаций с полнотекстовым поиском, фильтрацией по командам и доменам, метаданными о владельцах.

Интерактивная визуализация зависимостей — направленный граф «кто чей API вызывает» с фильтрами, масштабированием и экспортом в PNG/SVG.

Автоматический анализ критических изменений — сравнение двух версий контракта с классификацией изменений: критические (🔴), устаревшие (🟠), новые (🟢), изменённые (🟡).

Анализ влияния изменений — автоматическое определение всех сервисов, использующих изменяемые endpoints, со списком контактов владельцев.

Полная история версий — временная линия изменений с возможностью сравнения любых двух версий и отката к предыдущей.

Что не входит в область проекта​

  • Контрактное тестирование — автоматическая генерация тестов (инструменты типа Pact).
  • Управление шлюзом API — маршрутизация, rate limiting, аутентификация на уровне шлюза.
  • Мониторинг производительности — время отклика, пропускная способность (задача Prometheus/Grafana).
  • Замена сервисной сетки — система дополняет, но не заменяет Istio/Linkerd.
  • Генерация SDK — автоматическая генерация клиентских библиотек отложена до версии 4.0.

Объём последующих релизов​

Версия 2.0 (6–9 мес. после MVP) — интеграция с CI/CD (GitHub Actions, GitLab CI), API вебхуков, уведомления email/Slack, CLI-инструмент.

Версия 3.0 (12–18 мес.) — автоматическое обнаружение зависимостей, интеграция с трассировкой (Jaeger, Zipkin), контрактное тестирование.

Версия 4.0 (18+ мес.) — поддержка gRPC, GraphQL, AsyncAPI, генерация клиентских SDK.

Технологический стек​

Серверная часть​

Node.js + Express.js — выбран за нативную работу с JSON (OpenAPI-контракты), богатую экосистему npm для OpenAPI-инструментов (oasdiff), быструю разработку и поддержку async/await для неблокирующего анализа больших контрактов.

Клиентская часть​

React.js + React Flow — компонентная архитектура с переиспользуемыми компонентами (ServiceCard, ContractDiff), готовое решение для интерактивных графов с поддержкой масштабирования и перетаскивания.

База данных​

PostgreSQL с JSONB — реляционная структура для таблиц services, contracts, dependencies; тип JSONB для хранения OpenAPI-контрактов с GIN-индексами; открытый исходный код.
