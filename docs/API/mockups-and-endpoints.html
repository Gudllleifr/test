Макеты и эндпоинты

Шаблон

Этот файл является шаблоном. Вставьте макеты интерфейса в отмеченные места и дополните описание эндпоинтов.

Экраны пользовательского интерфейса​

MVP включает четыре основных экрана, соответствующих ключевым сценариям использования.

Экран 1 — Реестр сервисов​

Главный экран системы. Показывает список всех зарегистрированных сервисов с возможностью поиска и фильтрации.

Ключевые элементы:

  • поисковая строка (поиск по имени сервиса, ILIKE);
  • фильтры: по команде (owner_team), по домену (domain);
  • карточка сервиса: название, команда, домен, текущая версия контракта, количество endpoints, дата обновления;
  • кнопка «Зарегистрировать сервис».

Экран 2 — Страница сервиса​

Детальная информация о конкретном сервисе.

Ключевые элементы:

  • метаданные сервиса: название, домен, команда, email владельца, ссылка на репозиторий;
  • текущая активная версия контракта с кнопкой «Загрузить новую версию»;
  • вкладки: «История версий» (последние 10), «Зависимости» (входящие и исходящие);
  • кнопка «Сравнить версии».

Экран 3 — Сравнение версий контракта​

Результаты сравнения двух версий OpenAPI-контракта.

Ключевые элементы:

  • выпадающие списки для выбора базовой и целевой версии;
  • сводная панель: счётчики по категориям (🔴 критические, 🟠 устаревшие, 🟢 новые, 🟡 изменённые);
  • параллельное сравнение: две колонки с подсветкой изменений построчно;
  • для каждого критического изменения — пояснение последствий;
  • кнопка «Анализ влияния» (активна при наличии критических изменений).

Экран 4 — Граф зависимостей​

Интерактивный направленный граф всех сервисов и их зависимостей.

Ключевые элементы:

  • граф (React Flow): узлы = сервисы, стрелки = зависимости (от потребителя к поставщику);
  • всплывающая подсказка при наведении: название, команда, количество входящих/исходящих зависимостей;
  • клик на узел → переход на страницу сервиса;
  • фильтр по команде (скрывает несвязанные узлы);
  • кнопка «Добавить зависимость».

REST API — Описание эндпоинтов​

Ниже приведён обзор REST API. Полная интерактивная спецификация доступна в разделе OpenAPI-спецификация.

Сервисы​

МетодПутьОписаниеUC
GET/api/v1/servicesСписок сервисов с пагинацией и фильтрамиUC-003
POST/api/v1/servicesРегистрация нового сервисаUC-001
GET/api/v1/services/{id}Детальная информация о сервисеUC-001
PATCH/api/v1/services/{id}Обновление метаданных сервиса—
DELETE/api/v1/services/{id}Удаление сервиса (только Admin)—

Версии контрактов​

МетодПутьОписаниеUC
GET/api/v1/services/{id}/versionsИстория версий контракта (последние 10)UC-005
POST/api/v1/services/{id}/versionsЗагрузка новой версии контрактаUC-001 (A1)
GET/api/v1/services/{id}/versions/{versionId}Просмотр конкретной версии (метаданные + spec_content)UC-005
POST/api/v1/services/{id}/compareСравнение двух версий контрактаUC-002
POST/api/v1/services/{id}/impactАнализ влияния измененийUC-004

Зависимости​

МетодПутьОписаниеUC
GET/api/v1/graphГраф всех сервисов и зависимостейUC-003
POST/api/v1/dependenciesРегистрация зависимостиUC-003 (A1)
DELETE/api/v1/dependencies/{id}Удаление зависимости—

Аутентификация​

МетодПутьОписание
POST/api/v1/auth/loginВход в систему, получение JWT
POST/api/v1/auth/refreshОбновление access token по refresh token
POST/api/v1/auth/logoutВыход из системы

Соглашения API​

  • Формат ответа: JSON, кодировка UTF-8.
  • Аутентификация: Bearer JWT в заголовке Authorization.
  • Пагинация: параметры page и limit (по умолчанию limit=20, максимум limit=100).
  • Ошибки: формат {error: {code, message, details}} с HTTP-кодами 400, 401, 403, 404, 409, 500.
  • Версионирование: все эндпоинты начинаются с /api/v1/.
