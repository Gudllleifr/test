API docs by Redocly

API Contract Hub (1.0.0)

Download OpenAPI specification:Download

Централизованный реестр API-контрактов для микросервисной архитектуры ООО "ПринтСтройМонтажСервис" (120 сервисов, 15 команд).

Покрывает три экрана: реестр сервисов (/services), детали сервиса (/services/:id), граф зависимостей (/graph). Экран деталей включает встроенные блоки сравнения версий (UC-002) и анализа влияния (UC-004) (это не отдельные страницы).

services

Реестр сервисов, версии контрактов, сравнение, анализ влияния

Список сервисов (экран 1)

Постраничный список всех зарегистрированных сервисов с фильтрацией. Источник данных для таблицы и счётчиков на экране "Реестр сервисов"

Authorizations:
BearerAuth
query Parameters

integer >= 1
Default: 1
limit
integer [ 1 .. 100 ]
Default: 20

string
Example: search=payments

Поиск по названию, регистронезависимый (BR-001)

team

string (ServiceDomain)
Enum: "payments" "orders" "users" "logistics" "platform" "analytics" "other"
Example: domain=payments

Фильтр по домену

Зарегистрировать сервис (UC-001)

Создаёт сервис и сохраняет первый контракт (v1). Файл OpenAPI проходит валидацию: синтаксис, схема 2.0/3.0/3.1, наличие info.title и info.version. При ошибке валидации - 400, при дублировании имени - 409 (BR-001). Заголовок x-idempotency-key предотвращает дублирование при ретраях.

Authorizations:
BearerAuth
header Parameters
x-idempotency-key
string <uuid>

UUID для идемпотентности (повторный запрос с тем же ключом → тот же 201)

Request Body schema: multipart/form-data
required
name
required
string [ 3 .. 100 ] characters

Уникальное название, 3–100 символов, регистр не учитывается (BR-001)

domain
required
string (ServiceDomain)
Enum: "payments" "orders" "users" "logistics" "platform" "analytics" "other"

Функциональный домен сервиса

required
object (OwnerInfo)

Контактная информация владельца нужна для блока зависимостей и анализа влияния

repositoryUrl
string or null <uri>
contractFile
required
string <binary>

Файл .json или .yaml, максимум 5 МБ (BR-002)

Анализ влияния изменений (UC-004, встроен в экран 2)

Находит все сервисы с зарегистрированной входящей зависимостью от данного, показывает изменившиеся endpoints и контакты владельцев. Анализ основан только на вручную зарегистрированных зависимостях (BR-014). Система не знает, какие именно endpoints вызывает каждый потребитель, поэтому показываются все изменившиеся (BR-015).

Зарегистрировать зависимость (UC-003 поток A1)

Создаёт направленную связь: sourceServiceId (потребитель) → targetServiceId (поставщик). Стрелка в графе направлена от потребителя к поставщику (BR-010). В MVP зависимости регистрируются только вручную (BR-011).

BearerAuth
Request Body schema: application/json
required
sourceServiceId
required
string <uuid>

ID потребителя (кто будет вызывать API)

targetServiceId
