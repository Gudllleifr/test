## Сущности системы {#entities}

Анализ проводился по каждой сущности системы с учётом характера данных, объёма, паттернов доступа, требований к консистентности и стоимости.

| Сущность | Ключевые атрибуты | Акторы | Характер |
|---------|-----------------|--------|---------|
| `Service` | `id`, `name`, `domain`, `owner_team`, `owner_email`, `repository_url`, `created_at` | Разработчик, Архитектор — CRUD; Тех. руководитель — только чтение | Транзакционный |
| `ContractVersion` | `id`, `service_id`, `version_number`, `spec_content`, `spec_format`, `openapi_spec_version`, `endpoints_count`, `change_summary`, `is_active`, `is_archived` | Разработчик/Архитектор загружают; все роли читают; oasdiff обрабатывает `spec_content` | Транзакционный + аналитический |
| `Dependency` | `id`, `source_service_id`, `target_service_id`, `registered_by`, `registered_at` | Разработчик/Архитектор создают/удаляют; все читают для графа | Транзакционный |
| `User` | `id`, `username`, `email`, `password_hash`, `role`, `failed_login_count`, `locked_until` | Система читает при каждом JWT-запросе | Транзакционный |
| `AuditLog` | `id`, `user_id`, `action`, `entity_type`, `entity_id`, `ip_address`, `timestamp`, `metadata` | Система пишет автоматически | Аналитический |

## Сравнительный анализ по сущностям {#analysis}

<Tabs>
<TabItem value="service" label="Service" default>

| Критерий | Значение |
|---------|---------|
| Объём данных | ≤ 150 строк, < 1 МБ |
| Паттерн чтения/записи | OLTP. Чтение доминирует (~15 запросов/сессию). Запись редкая |
| Консистентность | Strong (ACID). Уникальность имени (BR-001), целостность FK с версиями |
| Транзакции | Нужны. Создание сервиса + создание v1 контракта — одна транзакция |
| Требования к запросам | Фильтрация по `team`, `domain`. Поиск по `name` (ILIKE). Пагинация. JOIN с `ContractVersion` |
| **Итого** | **PostgreSQL** |

</TabItem>
<TabItem value="contract" label="ContractVersion">

| Критерий | Значение |
|---------|---------|
| Объём данных | ≤ 1 500 строк. Контент: средний OpenAPI ~50–100 КБ → 75–150 МБ. Макс. 5 МБ/файл → до 7,5 ГБ. С учётом NFR-008 production: ~5 ГБ |
| Паттерн чтения/записи | Запись редкая (загрузка новой версии). Чтение: метаданные часто, `spec_content` при каждом сравнении целиком. OLTP + тяжёлое единичное чтение для oasdiff |
| Консистентность | Strong (ACID). NFR-005: атомарность операции сохранения. Откат при частичном сбое (BR-019) |
| Транзакции | Нужны. Сохранение контракта + обновление `current_version` в Service — атомарно |
| Требования к запросам | Выборка по `service_id` + `version_number`. Полный `spec_content` для oasdiff. История: ORDER BY + LIMIT 10 (BR-017). GIN-индекс на JSONB для `change_summary` |
| **Итого** | **PostgreSQL + JSONB/TEXT для `spec_content`** |

|---------|---------|
| Объём данных | ≤ 500 строк, < 1 МБ |

## Итоговые решения по хранению {#decisions}

| Сущность | Технология | Обоснование |
|---------|-----------|-------------|
| Service | PostgreSQL | Реляционные связи, жёсткая схема, OLTP |
| ContractVersion | PostgreSQL + JSONB/TEXT для `spec_content` | Метаданные — реляционно; контент — TEXT (атомарное чтение всего документа при сравнении) |
| Dependency | PostgreSQL | Реляционная структура, два FK-индекса, OLTP |
| User | PostgreSQL | ACID критично для безопасности |
| AuditLog | PostgreSQL (MVP) → Loki/ELK (production) | Write-heavy изоляция; eventual consistency достаточна |

## Предположение, требующее проверки {#assumption}

:::caution Важно проверить
Средний размер OpenAPI-файла принят 50–100 КБ на основе типичных enterprise-спецификаций. Если команды хранят подробные примеры запросов/ответов, реальный размер может достигать 500 КБ–1 МБ. Это меняет оценку объёма с ~150 МБ до ~1,5 ГБ за 1 500 версий.

**Для проверки:** собрать реальные OpenAPI-файлы хотя бы 10 существующих сервисов организации, посчитать среднее и перемножить на 1 500. Если среднее > 1 МБ — рассмотреть Object Storage (S3/MinIO) для `spec_content` с хранением только метаданных в PostgreSQL.
:::

## Общая схема БД {#schema}

```
PostgreSQL (основная БД)
├── users
├── services
├── contract_versions     (метаданные)
├── contract_version_content  (spec_content, горячее/холодное разделение)
├── dependencies
└── audit_logs            (партиционирование по месяцу)

Опционально v2.0:
├── Redis          → JWT blacklist, rate limiting по IP, кеш графа
└── Loki/ELK       → audit_logs (write-heavy изоляция из PostgreSQL)
```
