Документ содержит функциональные требования к первой версии продукта API Contract Hub (MVP v1.0). Предназначен для команды разработки, тестировщиков, архитектора и технических руководителей.

:::note Скоуп требований
Требования распространяются исключительно на MVP v1.0. Границы MVP определены в [Концепции и границах продукта](../1-overview/concept-and-scope).
:::

## Контекст системы {#context}

| Параметр | Значение |
|---------|---------|
| Организация | ООО «ПринтСтройМонтажСервис» |
| Масштаб | 120 микросервисов, 15 команд, ~110 пользователей |
| Платформа | Node.js + Express.js, React.js + React Flow, PostgreSQL |
| Развёртывание | Docker Compose (MVP), Kubernetes (production) |
| Основная проблема | 18 API-инцидентов/мес., 4 часа ручного анализа на каждое изменение |

| ID | Название | Акторы | Приоритет |
|----|---------|--------|----------|
| UC-001 | Регистрация нового сервиса и загрузка OpenAPI-контракта | Разработчик, Архитектор | Must Have |
| UC-002 | Сравнение версий контракта и обнаружение критических изменений | Разработчик, Архитектор | Must Have |
| UC-003 | Просмотр графа зависимостей сервисов | Разработчик, Архитектор, Тех. руководитель | Must Have |

| UC-005 | Просмотр истории версий контракта | Разработчик, Архитектор, QA-инженер | Should Have |

## UC-001 — Регистрация нового сервиса и загрузка контракта {#uc-001}

| Поле | Значение |
|------|---------|
| **ID** | UC-001 |
| **Акторы** | Разработчик серверной части (основной), Архитектор (дополнительный) |
| **Цель** | Зарегистрировать новый микросервис и загрузить его OpenAPI-спецификацию, чтобы другие команды могли видеть его контракт и зависимости |
| **Предусловия** | Пользователь аутентифицирован (роль: разработчик или архитектор); подготовлен файл OpenAPI (JSON или YAML, версия 2.0/3.0/3.1); сервис ещё не существует в реестре |

### Основной поток {#uc-001-main}

1. Пользователь переходит в «Реестр сервисов» и нажимает «Зарегистрировать сервис».
1. Система отображает форму регистрации.
1. Пользователь заполняет обязательные поля: название сервиса (3–100 символов), команда-владелец, контактный email, домен сервиса (payments, orders, users, logistics и др.) и опциональную ссылку на репозиторий.
1. Пользователь загружает OpenAPI-файл: через выбор файла с устройства (`.json` или `.yaml`, макс. 5 МБ) или вставку содержимого в текстовое поле.
1. Система выполняет валидацию: синтаксис (корректный JSON/YAML), соответствие схеме OpenAPI 2.0/3.0/3.1, наличие обязательных полей (`info.title`, `info.version`).
1. Система отображает краткое резюме распознанного контракта: название API, версия, количество endpoints.
1. Пользователь нажимает «Зарегистрировать».
1. Система сохраняет сервис и контракт, устанавливает версию контракта как `v1`.
1. Система перенаправляет на страницу детальной информации о сервисе.

### Альтернативный поток A1 — загрузка новой версии контракта {#uc-001-a1}

Пользователь открывает страницу существующего сервиса и нажимает «Загрузить новую версию контракта». Система пропускает форму регистрации и предлагает сразу загрузить файл. После валидации показывает автоматическое сравнение с предыдущей версией (вызов логики UC-002). После подтверждения сохраняет контракт как следующую версию (v2, v3 и т.д.).

### Исключительные потоки {#uc-001-exceptions}

**E1 — файл не прошёл валидацию:** система отображает список ошибок с указанием строки и типа ошибки (например: «Строка 42: поле `responses` обязательно для операции `GET /users`»). Сервис не сохраняется, пользователь остаётся на форме.

**E2 — имя сервиса занято:** система сообщает: «Сервис с именем [Имя] уже зарегистрирован. Если это ваш сервис, перейдите на его страницу для загрузки новой версии».

### Бизнес-правила {#uc-001-br}

| ID | Правило |
|----|---------|
| **BR-001** | Имя сервиса уникально в рамках системы, регистр не учитывается (`USERS-SERVICE` = `users-service`) |
| **BR-002** | Максимальный размер загружаемого файла — 5 МБ |
| **BR-003** | Поддерживаемые версии: OpenAPI 2.0 (Swagger), 3.0.x, 3.1.x |
| **BR-004** | Версия контракта назначается автоматически (v1, v2, …vN). Ручная установка версии не предусмотрена |
| **BR-005** | История версий хранит последние 10 версий контракта (ограничение MVP) |

## UC-002 — Сравнение версий контракта {#uc-002}

| Поле | Значение |
|------|---------|
| **ID** | UC-002 |
| **Акторы** | Разработчик (основной), Архитектор (дополнительный) |
| **Цель** | Получить детальный отчёт о различиях между двумя версиями контракта с классификацией изменений, чтобы оценить безопасность изменений API до деплоя |
| **Предусловия** | Пользователь аутентифицирован; для сервиса существует минимум 2 версии контракта |

### Основной поток {#uc-002-main}

1. Пользователь открывает страницу сервиса и переходит в «История версий».
1. Пользователь выбирает «Базовую» (старую) и «Целевую» (новую) версию из выпадающих списков.
1. Пользователь нажимает «Сравнить».
1. Система выполняет анализ через библиотеку `oasdiff`: сравниваются схемы, endpoints, параметры, типы данных, коды ответов.
1. Система классифицирует изменения: 🔴 критические (удаление endpoint, обязательного параметра, изменение типа), 🟠 устаревшие (`deprecated`), 🟢 новые (новые endpoints, необязательные параметры), 🟡 изменённые (описания, примеры).
1. Система отображает: сводную панель с количеством изменений по категориям и параллельное сравнение двух колонок с подсветкой.

### Альтернативный поток A1 — сравнение при загрузке {#uc-002-a1}

При загрузке новой версии (поток A1 в UC-001) система автоматически запускает сравнение с предыдущей версией. Если обнаружены критические изменения, показывает предупреждение: «Обнаружено N критических изменений. Они могут нарушить работу зависимых сервисов. Продолжить сохранение?»

### Бизнес-правила {#uc-002-br}

| ID | Правило |
|----|---------|
| **BR-006** | Изменения рассматриваются относительно «Базовой» (старой) версии |
| **BR-007** | Классификация критических изменений выполняется по правилам `oasdiff`. Версия библиотеки зафиксирована в `package.json` и обновляется только после проверки совместимости |
| **BR-008** | Одновременно сравниваются только 2 версии. Сравнение 3+ версий — вне скоупа MVP |
| **BR-009** | Результаты сравнения не кэшируются в MVP — каждый запрос выполняет новый анализ |

## UC-003 — Просмотр графа зависимостей {#uc-003}

| Поле | Значение |
|------|---------|
| **ID** | UC-003 |
| **Акторы** | Разработчик, Архитектор, Технический руководитель |
| **Цель** | Получить визуальное представление зависимостей между микросервисами, чтобы быстро понять архитектуру, найти критические пути и оценить масштаб влияния изменений |
| **Предусловия** | В системе зарегистрировано минимум 2 сервиса с хотя бы одной зависимостью |

### Основной поток {#uc-003-main}

1. Пользователь переходит в «Граф зависимостей» из главного меню.
1. Система загружает данные обо всех сервисах и зависимостях из БД.
1. Система рендерит интерактивный направленный граф: каждый сервис — узел (название + команда), зависимость «A использует API B» — стрелка от A к B.
1. Пользователь взаимодействует с графом: масштабирование (колесо мыши), панорамирование (перетаскивание), наведение на узел (всплывающая подсказка), клик на узел (переход на страницу сервиса).
1. Пользователь применяет фильтр по команде — граф показывает только её сервисы и зависимости.

### Бизнес-правила {#uc-003-br}

| ID | Правило |
|----|---------|
| **BR-010** | Граф направленный. Стрелка от «потребителя» к «поставщику» API |
| **BR-011** | В MVP зависимости регистрируются только вручную (автоматическое обнаружение — версия 3.0) |
| **BR-012** | Один сервис может иметь неограниченное количество входящих и исходящих зависимостей |
| **BR-013** | Граф отображает все сервисы, включая не имеющие зависимостей (изолированные узлы) |

## UC-004 — Анализ влияния изменений {#uc-004}

| Поле | Значение |
|------|---------|
| **ID** | UC-004 |
| **Акторы** | Разработчик (основной), Архитектор (дополнительный) |
| **Цель** | Определить, какие сервисы пострадают от планируемых изменений API, получить контакты их владельцев и оценить масштаб влияния |
| **Предусловия** | Существует сервис с минимум 2 версиями контракта; зарегистрированы входящие зависимости |

### Основной поток {#uc-004-main}

1. Пользователь находится на экране результатов сравнения (после UC-002) или на странице сервиса.
1. Пользователь нажимает «Анализ влияния» (доступна при наличии критических или изменённых endpoints).
1. Система определяет список изменившихся endpoints из результатов сравнения.
1. Система находит все сервисы, у которых в зависимостях указан целевой сервис.
1. Система формирует отчёт: для каждого зависимого сервиса — название, команда-владелец, контактный email, перечень изменённых endpoints, тип изменения.
1. Система показывает сводную статистику: «Затронуто N сервисов, из них M с критическими изменениями».
1. Пользователь может скопировать список email-адресов владельцев для оповещения.

### Бизнес-правила {#uc-004-br}

| ID | Правило |
|----|---------|
| **BR-014** | Анализ влияния основывается только на вручную зарегистрированных зависимостях (MVP). Точность ограничена полнотой данных |
| **BR-015** | В MVP система не выполняет автоматическую оценку критичности для каждого зависимого сервиса — показывается только факт использования изменившихся endpoints. Это должно быть явно отражено в UI |

## UC-005 — Просмотр истории версий контракта {#uc-005}

| Поле | Значение |
|------|---------|
| **ID** | UC-005 |
| **Акторы** | Разработчик (основной), Архитектор (дополнительный), QA-инженер (только просмотр) |
| **Цель** | Получить хронологический список всех версий контракта с возможностью просмотра, сравнения и отката |
| **Предусловия** | Для выбранного сервиса существует минимум 1 версия контракта |

### Основной поток {#uc-005-main}

1. Пользователь открывает страницу сервиса и переходит на вкладку «История версий».
1. Система отображает таблицу версий в обратном хронологическом порядке (новые — сверху): номер версии, дата и время загрузки, имя пользователя, краткая сводка изменений относительно предыдущей версии.
1. Пользователь нажимает «Просмотреть» — система открывает полный текст спецификации с подсветкой синтаксиса.
1. Пользователь выбирает две версии и нажимает «Сравнить» — запускается сценарий UC-002.

### Альтернативный поток A1 — откат к предыдущей версии {#uc-005-a1}

Доступен только для роли «Архитектор». Пользователь нажимает «Сделать активной» для одной из версий. Система показывает предупреждение: «Это действие создаст новую версию vM с содержимым vN. Продолжить?» После подтверждения — создаётся новая версия, оригинальная история остаётся неизменной.

### Бизнес-правила {#uc-005-br}

| ID | Правило |
|----|---------|
| **BR-017** | История ограничена последними 10 версиями. При добавлении 11-й самая старая архивируется (метаданные хранятся, но версия недоступна для сравнения) |
| **BR-018** | Удаление версии контракта не предусмотрено в MVP |
| **BR-019** | Откат создаёт новую версию — история версий неизменна |
