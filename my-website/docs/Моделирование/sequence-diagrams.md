Sequence-диаграммы описывают взаимодействие компонентов системы во времени для ключевых сценариев MVP.

## Загрузка контракта и обнаружение изменений {#seq-upload}

Диаграмма охватывает полный поток: загрузка файла → валидация → параллельное сравнение через `oasdiff` → предупреждение при критических изменениях → сохранение с управлением историей.

```plantuml
@startuml SD_Upload_Contract
actor "Разработчик /\nАрхитектор" as user
participant "Web UI" as ui
participant "API Contract Hub\n(Backend)" as backend
participant "oasdiff" as oasdiff
database "PostgreSQL" as db

user -> ui : Нажать «Загрузить версию контракта»
ui -> user : Отобразить форму загрузки

user -> ui : Загрузить файл / вставить текст
ui -> backend : POST /contracts/upload (file, serviceId)

group Валидация
  alt Файл > 5 МБ
    backend -> ui : 400 Файл превышает 5 МБ
    ui -> user : Сообщение об ошибке
  else Синтаксис не JSON/YAML
    backend -> ui : 400 Ошибки синтаксиса (строка + тип)
    ui -> user : Список ошибок валидации
  else Не OpenAPI 2.0/3.0/3.1
    backend -> ui : 400 Не соответствует схеме OpenAPI
    ui -> user : Список ошибок валидации
  else Валидация пройдена
    backend -> backend : Сформировать резюме контракта
  end
end

par Параллельная обработка
  backend -> oasdiff : Запустить сравнение с предыдущей версией
  oasdiff -> backend : Список изменений (breaking/new/modified)
also
  backend -> backend : Сформировать резюме контракта
end

backend -> backend : Классифицировать изменения

opt Обнаружены критические изменения
  backend -> ui : Предупреждение о критических изменениях
  ui -> user : Показать диалог с деталями
  alt Пользователь нажал «Отмена»
    user -> ui : Отмена
    ui -> user : Вернуться на страницу сервиса
  else Пользователь нажал «Сохранить»
    user -> ui : Подтвердить сохранение
  end
end

user -> ui : Нажать «Сохранить»
ui -> backend : POST /contracts/save

backend -> db : Сохранить контракт, назначить версию vN

alt Версий <= 10
  backend -> db : Создать запись в истории версий
else Версий > 10
  backend -> db : Архивировать самую старую версию
  backend -> db : Создать запись в истории версий
end

backend -> ui : Страница сравнения версий (diff)
ui -> user : Отобразить результаты сравнения

@enduml
```

Ключевые архитектурные решения, отражённые в диаграмме:

- Параллельная обработка (`par`) — `oasdiff` запускается одновременно с формированием резюме контракта, что укладывается в NFR-001 (≤ 3 с для сравнения).
- Управление историей через `alt` — архивирование самой старой версии при превышении лимита в 10 версий реализует BR-017 прямо на уровне сохранения.
- Диалог подтверждения (`opt`) активируется только при критических изменениях — не блокирует разработчика при безопасных правках.
