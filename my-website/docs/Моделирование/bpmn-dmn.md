В этом разделе описаны бизнес-процессы системы в нотации BPMN 2.0 и правила принятия решений в нотации DMN.

## Процесс 1 — Загрузка контракта и обнаружение изменений {#bpmn-upload}

Процесс охватывает сценарии UC-001 (поток A1) и UC-002: разработчик загружает новую версию OpenAPI-контракта, система автоматически сравнивает её с предыдущей и классифицирует изменения.

**Участники процесса:** Разработчик, Система (API Contract Hub), oasdiff (библиотека анализа).

**Описание процесса:**

Разработчик инициирует загрузку новой версии контракта с существующей страницы сервиса. Система валидирует файл: при ошибке возвращает список проблем и завершает процесс без сохранения. При успешной валидации oasdiff сравнивает новую версию с текущей активной, классифицирует изменения и возвращает сводку. Если обнаружены критические изменения, пользователю показывается предупреждение с запросом подтверждения. После подтверждения система сохраняет версию, обновляет `change_summary` и делает версию активной.

## Процесс 2 — Анализ влияния изменений {#bpmn-impact}

Процесс охватывает сценарий UC-004: пользователь получает список затронутых сервисов после выявления критических изменений.

**Участники процесса:** Разработчик / Архитектор, Система (API Contract Hub).

<!-- 
  ВСТАВЬТЕ BPMN-ДИАГРАММУ ЗДЕСЬ.
-->

**Описание процесса:**

Пользователь инициирует анализ влияния с экрана результатов сравнения. Система извлекает список изменившихся endpoints и выполняет обратный поиск в таблице зависимостей: находит все сервисы, у которых в зарегистрированных зависимостях указан целевой сервис. Система формирует отчёт с контактами владельцев. Если зависимые сервисы не найдены — показывает информационное сообщение и предлагает перейти к графу зависимостей.

## Правило принятия решений — Классификация изменений {#dmn-classification}

DMN-таблица описывает логику классификации изменений, которую выполняет `oasdiff`.

<!-- 
  ВСТАВЬТЕ DMN-ТАБЛИЦУ ЗДЕСЬ (draw.io или ссылка на Camunda DMN editor).
-->

| Тип изменения | Примеры | Классификация | Цвет в UI |
|--------------|---------|--------------|----------|
| Удаление endpoint | `DELETE /v1/orders` | Критическое | 🔴 |
| Удаление обязательного параметра | Убран обязательный `order_id` | Критическое | 🔴 |
| Изменение типа данных | `string` → `integer` | Критическое | 🔴 |
| Изменение кода ответа | `200` → `201` | Критическое | 🔴 |
| Пометка как устаревший | `deprecated: true` | Устаревшее | 🟠 |
| Добавление нового endpoint | `POST /v1/shipments` | Новое | 🟢 |
| Добавление необязательного параметра | Добавлен опциональный `metadata` | Новое | 🟢 |
| Изменение описания | Изменён текст `description` | Изменённое | 🟡 |
| Изменение примера | Изменён `example` в схеме | Изменённое | 🟡 |

:::tip Совет
Правила классификации берутся из [oasdiff changelog rules](https://github.com/tufin/oasdiff). Версия библиотеки зафиксирована в `package.json` согласно BR-007.
:::
