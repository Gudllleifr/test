## Стейкхолдеры {#stakeholders}

Стейкхолдеры — все лица и группы, которые оказывают влияние на проект или испытывают его влияние на себе.

| Стейкхолдер | Тип влияния | Уровень интереса | Ожидания | Формат взаимодействия |
|-------------|------------|-----------------|---------|----------------------|
| Руководство ООО (CTO, VP Engineering) | Финансирование, приоритеты | Высокий | ROI от снижения инцидентов | Ежемесячные статус-отчёты |
| Технические руководители (~10 чел.) | Приоритизация задач команд | Высокий | Координация изменений, сводный дашборд | Демо каждые 2 недели |
| Архитекторы решений (~8 чел.) | Технические решения и стандарты | Высокий | Граф зависимостей, аудиторский след, точность >95% | Ключевые пользователи бета-версии |
| Разработчики серверной части (~80 чел.) | Основные пользователи | Средний | Быстрый анализ влияния, простой UI, онбординг `≤30 мин.` | Пилот с 3 командами |
| Инженеры DevOps (~12 чел.) | Интеграция в CI/CD | Средний | REST API, развёртывание `≤30 мин.`, `≤2 CPU / 4 GB RAM` | Техническая интеграция на пилоте |

## Целевая аудитория {#audience}

| Группа | Основная ценность | Отношение к системе | Ключевые ограничения |
|--------|------------------|--------------------|--------------------|
| Разработчики серверной части (~80 чел.) | Уверенность в изменениях API, экономия 3–4 ч/нед. | Заинтересованы, но скептичны | Максимум `30 мин.` на онбординг |
| Архитекторы решений (~8 чел.) | Контроль архитектуры, визуализация всей системы | Сильно поддерживают | Требуют точности графа >95% |
| Инженеры DevOps (~12 чел.) | Автоматизация проверок, снижение инцидентов | Поддерживают, если легко интегрируется | Max `2 CPU`, `4 GB RAM`, развёртывание `<30 мин.` |
| Технические руководители (~10 чел.) | Координация между командами, оценка рисков | Поддерживают | Нужен сводный вид без лишних деталей |

Проект реализуется силами штатных сотрудников без привлечения аутсорса.

## Бизнес-допущения и зависимости {#assumptions}

**Критические допущения:**

- Все микросервисы используют или могут быть переведены на OpenAPI 3.0 для документирования API.
- Команды разработки готовы выделять 2–3 часа в неделю на регистрацию сервисов в первые 3 месяца.
- Руководство поддерживает инициативу и выделяет ресурсы DevOps для технической интеграции.
- Существующая инфраструктура поддерживает развёртывание новых сервисов (Kubernetes-кластер с доступными ресурсами).
- Средняя частота изменений API: 2–3 критических изменения на сервис в год.

**Внешние зависимости:**

- Библиотеки Open Source: `oasdiff` (MIT), `react-flow` (MIT), PostgreSQL.
- Облачная инфраструктура: Docker для контейнеризации, Kubernetes для оркестрации.
- Существующие системы: GitHub/GitLab для хранения кода, PagerDuty для управления инцидентами.
